# Privacy and Permissions

Last reviewed: 2026-06-27

This document records the production privacy and permissions contract for RelateAI. It is not legal advice; the Play Console release owner must verify every declaration against the final build, store listing, privacy policy, and current Google Play policy before release.

## WhatsApp Accessibility Contract

Current implementation:

- Service: `com.example.core.accessibility.WhatsAppAccessibilityService`.
- Manifest permission: `android.permission.BIND_ACCESSIBILITY_SERVICE`.
- Service metadata: `app/src/main/res/xml/accessibility_service_config.xml`.
- Package scope: `com.whatsapp` and `com.whatsapp.w4b` only.
- App-level consent: stored in `SecurePrefs`, shown in AI Doctor, and enforced before `WhatsAppSender` can be called.
- Failure taxonomy: `WhatsAppSendResult` maps service-disabled, locked-device, app-not-found, compose-field, text-verification, send-button, and timeout failures to redacted provider metadata.

Required behavior:

- The channel must remain opt-in and revocable.
- Missing app-level consent must block WhatsApp sender invocation and surface as a setup issue.
- The service must not read, store, export, or log WhatsApp chat contents.
- The service may inspect only the current WhatsApp UI state required to find compose/send controls and verify the approved text before tapping send.
- Provider failure logs must remain redacted and must not include message body or screen contents.
- Fallback routes may run only through the existing dispatch eligibility and route-selection policies.

Policy risk:

- RelateAI's WhatsApp service is automation for message sending, not a disability-assistive accessibility tool. Do not mark the service as `isAccessibilityTool=true` unless the product purpose changes and a separate accessibility-tool review is completed.
- Google Play's AccessibilityService API policy requires automation to be narrow and clearly understood, and it prohibits autonomous Accessibility API actions that independently initiate, plan, and execute decisions. The release owner must verify that every WhatsApp send follows a static user-defined dispatch rule and does not let AI independently decide who to message, what to say, and when to send without the app's approval policies.
- Apps that are not accessibility tools need a separate prominent disclosure and affirmative consent for Accessibility API use. The current AI Doctor checkbox is a code-level consent gate; release review must still validate whether the in-app disclosure placement and copy satisfy Play's prominent-disclosure requirements.

Release evidence required:

- Screenshot or video of the in-app WhatsApp automation disclosure and affirmative checkbox.
- Screenshot or video of the Android Accessibility settings enablement flow for `RelateAI - Auto WhatsApp`.
- Screenshot or video showing consent denial leaves the app usable and keeps WhatsApp sends blocked.
- Screenshot or test evidence showing the no-send AI Doctor dry run reports missing consent/accessibility setup without sending a message.
- Evidence that the service remains scoped to WhatsApp package names and does not persist screen text.

## Data Safety Inputs

Data types currently in scope for the Play Data Safety form and privacy policy review:

- Contact info: names, phone numbers, email addresses, birthdays/events, relationship labels, notes, and personalization metadata.
- User-generated content: message drafts, approved messages, feedback, history, backup files, and manual contact/event entries.
- Account/auth data: Firebase/Google auth identifiers and OAuth/session state.
- App activity/diagnostics: dispatch attempts, delivery state, setup checks, recovery diagnostics, and non-sensitive analytics summaries.
- AI data: prompts and generated responses may include contact/event/message context. They must be disclosed as sent to the AI provider if enabled.
- Email/SMS send data: recipient address/number, subject where applicable, and body.
- WhatsApp automation data: approved outgoing message text and transient WhatsApp UI control inspection. Current contract says no WhatsApp chat content is collected or stored.

Sensitive data handling requirements:

- Auto backup remains disabled or sensitive stores remain excluded.
- User backup/export must be explicit, encrypted where implemented, and documented with restore limitations.
- Sign-out must clear local data through the single auth-layer orchestrator.
- Logs, analytics exports, backup manifests, and provider failure metadata must redact tokens, credentials, raw AI responses, raw screen contents, and message bodies where not explicitly user-exported.

## Open Release Blockers

- A release owner must complete the Play Accessibility declaration review for WhatsApp automation.
- A release owner must validate whether the current WhatsApp consent UI is sufficiently prominent or whether a dedicated disclosure screen is required before Accessibility enablement.
- A release owner must complete Data Safety and privacy policy copy against the final shipped feature set.
- If WhatsApp Accessibility automation is not approved for Play distribution, the release plan must document whether the production Play build disables that channel or whether distribution is limited outside Play.
